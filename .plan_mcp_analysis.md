# MCP 桥接问题分析报告

## 问题分类

### P0: ask_user 不触发弹窗
- **原项目行为**: `do_ask_user` 返回 `should_exit=True`，引擎 break 退出循环，等待用户外部输入
- **MCP 行为**: 之前返回普通 JSON，模型将其当作工具输出，自行选择第一项继续
- **修复**: 使用 MCP SDK 的 `session.elicit_form()` API（已修复）

### P1: 工具执行后无工作记忆上下文注入
- **原项目行为**: 几乎每个工具执行完后都注入 `_get_anchor_prompt()`（含 history/key_info/related_sop）
- **MCP 行为**: 完全缺失，返回只是普通 JSON
- **影响**: Trae 模型每轮执行工具后看不到工作记忆上下文，可能导致推理断链
- **缓解**: MCP 有 `ga_end_turn` + `ga_get_context` 手动工具，但依赖 Trae 主动调用
- **建议**: 在 `.trae/rules/genericagent-mcp.md` 中强化引导 Trae 在关键节点调用 lifecycle 工具

### P1: turn_end_callback 自动逻辑缺失
- **原项目**: 引擎每轮自动调用，注入警告/记忆/计划提示
- **MCP**: 需手动调用 `ga_end_turn`
- **子问题**:
  - Plan 70 轮超时强制 ask_user：MCP 完全缺失
  - 外部文件注入 `_keyinfo`/`_intervene`：MCP 完全缺失
  - 钩子扩展：MCP 完全缺失

### P2: do_no_tool 引擎自主逻辑缺失
- 空回复重试、回复截断处理、Plan 完成拦截、大代码块检测等
- MCP 架构下由 Trae 自行处理更合理，不属于 bug

### P3: 数据处理差异
- `no_tool` 数据不进入 tool_results 等
- 影响较小

## 建议修复清单

1. ✅ P0: ask_user 使用 elicit_form (已完成)
2. P1: 在 `.trae/rules` 中强化 lifecycle 调用引导
3. P2: 在 `ga_end_turn` 中添加 Plan 70 轮超时逻辑
4. P2: 考虑添加外部文件注入机制（可选）
