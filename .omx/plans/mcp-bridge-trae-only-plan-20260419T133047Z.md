# Revised Plan: Trae-only MCP Bridge Parity (No ga_authoritative)

## Change from previous plan

User constraint update: remove `ga_authoritative` / GenericAgent-owned LLM loop entirely. GenericAgent must not call its own LLM backend and must not own the agent loop. Allowed paths only:

1. Trae model calls GenericAgent MCP tools/resources/prompts.
2. Optional Trae/IDE hook or extension layer, if feasible, injects protocol/context or calls MCP helper tools.

## Consequence

Full original GenericAgent parity is impossible under this constraint. The bridge can provide state, context, validation, warnings, plan gates, scheduler due signals, and audit, but cannot force Trae to call tools or continue turns unless Trae exposes suitable host hooks.

## Target modes

### Mode A: `trae_mcp_contract` — baseline, must implement first

Loop owner: Trae.

GenericAgent MCP provides:
- session-scoped state
- lifecycle tools
- context retrieval tools
- plan state/gates
- scheduler due tools/resources
- audit and lifecycle warnings
- `.trae/project_rules.md` protocol text to bias Trae behavior

No guarantee that Trae will call lifecycle tools.

### Mode B: `trae_hook_assisted` — optional if Trae extension/hook is feasible

Loop owner: Trae.

A Trae extension or hook-like companion does one or more of:
- ensures project rules are installed/updated
- watches files/session state
- provides commands/status UI to call lifecycle MCP tools
- uses Trae AI API chat/generate APIs to create assisted flows with injected context
- potentially wraps a custom command such as “GenericAgent: Continue task” that calls `get_context` before sending a prompt to Trae AI API

This still does not equal original `agent_runner_loop` unless the extension can intercept every user prompt, every model response, every tool call, and every stop event. Current public docs found do not clearly show those lifecycle interception hooks.

## Architecture

### 1. MCP lifecycle tools

Implement tools only; no internal LLM loop:

- `ga_begin_task(task, session_id?, resume=false, plan_required=false)`
- `ga_get_context(session_id?, scope='all')`
- `ga_update_working_checkpoint(session_id, key_info?, related_sop?, merge=true)`
- `ga_end_turn(session_id, assistant_summary?, tool_calls?, tool_results?, response_text?)`
- `ga_enter_plan_mode(session_id, plan_path, require_approval=true)`
- `ga_get_plan_status(session_id)`
- `ga_approve_plan(session_id, approved_by, verdict_ref?)`
- `ga_end_task(session_id, final_summary, verification_ref?, force=false)`
- `ga_list_sessions`, `ga_get_session_timeline`, `ga_get_lifecycle_warnings`
- `ga_get_scheduler_due`, `ga_mark_scheduler_done`

Use `ga_` prefix so Trae tool selection sees these as a coherent lifecycle group.

### 2. Tool-return continuation envelope

Every lifecycle and mutating tool returns:

```json
{
  "status": "ok|continue|blocked|warning|error",
  "session_id": "...",
  "phase": "executing",
  "turn": 3,
  "recommended_next_tool": "ga_end_turn",
  "recommended_next_prompt": "...",
  "must_read_before_next_action": "ga_get_context",
  "warnings": [],
  "plan_status": {"remaining_unchecked": 2, "can_complete": false}
}
```

This is advisory, but makes the desired next action explicit.

### 3. Context exposure strategy

Because many MCP clients support tools more reliably than prompts/resources, expose context through tools first:

- Primary: `ga_get_context`
- Secondary/progressive: MCP resources/prompts if Trae supports them
- Tertiary: `.trae/project_rules.md` generated by `ga_install_rules`

### 4. `.trae/project_rules.md` protocol

Generate/maintain rules such as:

```md
# GenericAgent MCP Protocol

Before using GenericAgent tools for a non-trivial task:
1. Call `ga_begin_task`.
2. Call `ga_get_context` before planning or continuing.
3. After a batch of tool calls, call `ga_end_turn` with a concise summary.
4. If `ga_end_turn` returns `recommended_next_prompt`, continue using that prompt.
5. Before claiming completion in plan mode, call `ga_get_plan_status` and verify remaining_unchecked=0 plus verification evidence.
6. Use `ga_update_working_checkpoint` whenever key constraints or discoveries must persist.
```

This is probably the strongest low-effort way to bias Trae because public Trae materials describe `.rules` as loaded during initialization and used as behavior/context rules.

### 5. Hook / extension feasibility analysis

Possible hook-assisted implementation levels:

#### H0: No extension, MCP + project rules only
- Feasibility: high
- Difficulty: low
- Guarantees: weak/advisory
- Implement now.

#### H1: Trae extension command + status UI
- Feasibility: medium-high if Trae extension APIs are VS Code-like.
- Difficulty: medium.
- What it can do:
  - install/update `.trae/project_rules.md`
  - show active GenericAgent session status
  - provide commands: Begin Task, Get Context, End Turn, Check Plan, Scheduler Due
  - watch workspace files and session files
- What it cannot guarantee:
  - cannot intercept all AI chat turns unless API exposes chat lifecycle events.

#### H2: Extension uses Trae AI API to start wrapped chat flows
- Feasibility: medium, based on public AI API examples that include `aiProvider.chat(request)`.
- Difficulty: medium-high.
- What it can do:
  - command “GenericAgent: Continue with context” obtains MCP context then calls Trae AI API `chat` with injected context.
  - can approximate `_get_anchor_prompt` for extension-initiated chats.
- Risks:
  - may create a separate conversation path from Trae’s normal chat UI.
  - tool calling from that extension-initiated chat may not expose MCP tool execution hooks.

#### H3: True lifecycle hook: before prompt / after tool / after response / stop
- Feasibility: currently unproven from found public docs.
- Difficulty: high to unknown.
- Required host APIs:
  - onUserPromptSubmit
  - beforeModelCall / prompt transform
  - afterModelResponse
  - beforeToolUse / afterToolUse
  - onStop / onSessionEnd
  - ability to mutate or append chat context
- If Trae lacks these APIs, cannot implement.

#### H4: UI automation / log scraping hook
- Feasibility: possible but not recommended.
- Difficulty: high.
- Fragility/security risk: high.
- Use only as research prototype, not product design.

## Revised feasibility verdict

Implementable now:
- session state and lifecycle MCP tools
- plan status/gates as MCP tools
- context retrieval via tool
- scheduler due detection as tool/resource
- `.trae/project_rules.md` protocol installer
- observability/timeline/warnings

Likely implementable with extension work:
- commands/status UI/file watchers/rules management
- extension-initiated “continue with context” flow using AI chat API if available in target Trae build

Not currently provable without Trae-specific private/official API confirmation:
- automatic before/after AI chat hooks
- automatic post-tool callback hook
- automatic prompt injection into Trae’s normal agent loop
- guaranteed tool sequencing

## Implementation order under new constraint

1. Rename previous authority model to only `trae_mcp_contract` and optional `trae_hook_assisted`.
2. Implement session-scoped state and MCP lifecycle tools.
3. Add continuation envelope to all relevant tool outputs.
4. Add `ga_install_rules` to write/update `.trae/project_rules.md` with protocol instructions.
5. Add docs for Trae usage: “For complex tasks, tell Trae: use GenericAgent MCP protocol.”
6. Add simulated-Trae tests for compliant/non-compliant tool calls.
7. Prototype Trae extension:
   - command palette commands
   - status bar/session view
   - file watcher for `.genericagent_mcp/sessions`
   - optional AI API `chat` wrapper if available.
8. Only after prototype, decide whether true chat lifecycle hooks exist. If not, document H3 as unsupported.

## Non-goals

- No `execute_task` that calls GenericAgent’s own LLM.
- No `agent_loop` tool that owns model turns.
- No scheduler daemon that executes LLM tasks independently of Trae.
- No claim of guaranteed automatic memory injection unless a verified Trae hook supports prompt transformation.

## Acceptance criteria

- `ga_begin_task` creates session and returns next recommended tool.
- `ga_get_context` mirrors original anchor prompt content as retrievable context.
- `ga_end_turn` updates history/turn/warnings when Trae calls it.
- Missing lifecycle calls are detectable only when subsequent calls expose a gap; docs state limitations.
- `ga_get_plan_status` blocks/warns completion claims through tool result, but docs state Trae may ignore it.
- `ga_install_rules` writes project rules and is idempotent.
- Extension prototype documents which Trae APIs exist in the target install and whether H2/H3 are feasible.
